# UX/UI бриф — Human Compatibility OS

Документ для дизайн-передачи (Figma / дизайн-сессия). Цель — улучшить UI/UX,
**не нарушая этическую доктрину продукта**. Раздел «Жёсткие ограничения» —
не предмет переосмысления, это часть продукта.

## 1. О продукте (контекст для дизайна)

Операционная HR-платформа «Fabrika Sredy / Среда важнее героизма». Измеряет
**давление среды** (выгорание, перегрузку, коммуникационный шум, нестабильность
лидерства) и совместимость человека со средой. Тон — **спокойный,
поддерживающий, не осуждающий**. Это не «оценка людей» и не медицинский диагноз,
а сигнальный слой для **проверки человеком**.

Эмоциональная задача интерфейса: снизить тревогу, а не нагнетать. Человек,
увидевший «высокий риск», не должен почувствовать приговор.

## 2. Жёсткие ограничения (НЕ менять)

1. **Палитра спокойная. Высокий риск — приглушённый оранжевый/янтарный, НИКОГДА
   не агрессивный красный.** (Зашито в `lib/risk.ts`.) Светофор мягкий:
   low = emerald, medium = amber, high = **orange (не red)**.
2. **Постоянные дисклеймеры** на каждом экране индивидуального результата:
   «не диагноз, не оценка личности, не основание для кадрового решения; решение
   принимает человек». Нельзя прятать/убирать (`RESULT_DISCLAIMER`,
   `NO_DECISION_DISCLAIMER`).
3. **Explainability-first**: результат всегда показывается с разбором по
   компонентам и интерпретацией, а не «голым» числом-вердиктом.
4. **Анонимизация**: если в выборке < 3 человек — данные скрыты, показывается
   нейтральная плашка. Это поведение UI, его надо сохранить и красиво оформить.
5. **Нет «решающих» элементов**: никаких кнопок «отклонить / уволить», авто-
   вердиктов, «скоров пригодности человека». Подбор — «совместимость со средой»,
   с advisory-подсказкой, решение за человеком.
6. UI на русском. Тёмная тема.

## 3. Текущие дизайн-токены (как есть)

| Токен | Значение | Где |
|---|---|---|
| Фон | `#0b0f1a` | `app/globals.css` `--background` |
| Текст | `#e8edf5` | `--foreground` |
| Шрифт | system-ui / Segoe UI / Roboto | globals.css (системный, без кастомного) |
| Поверхности | `bg-white/5`, границы `border-white/10` | Tailwind-утилиты по экранам |
| Радиусы | `rounded-lg` / `rounded-xl` | повсеместно |
| Риск (UI) | low `emerald-400`, medium `amber-400`, high `orange-400` (muted) | `lib/risk.ts` |
| Бренд-индиго | `#2e2e6e` / `#16183a` | логотип (`app/icon.svg`, `public/hcos_logo.svg`) |
| Overall-риск (подбор) | green / yellow / orange | hiring |

⚠️ **Нестыковка для дизайнера:** в `tailwind.config.ts` объявлены `risk.low/medium/high`
= `#16a34a / #d97706 / #dc2626` (включая **красный** high), но реально UI берёт
приглушённые классы из `lib/risk.ts`. При нормализации токенов — привести к
одной палитре (без красного), это удобная точка для дизайн-системы.

Сейчас **нет UI-кит-библиотеки и компонентов** — всё на сырых Tailwind-классах,
дублируется по страницам (карточки, кнопки, селекты, светофор, дисклеймер).
Это главная возможность: собрать **дизайн-систему/компоненты**.

## 4. Информационная архитектура

Глобальная шапка (`app/layout.tsx`): логотип + название → ссылки на разделы по
роли. 9 экранов:

| Экран | Путь | Для кого | Назначение |
|---|---|---|---|
| Главная | `/` | все | вход, 4 блока-промо, навигация |
| Логин | `/login` | все | email + пароль |
| Опросник | `/questionnaire` | сотрудник | выбор сессии (short/base/deep), шкала 1–5, согласие, результат + интерпретация |
| Дашборд | `/dashboard` | hr/lead/admin/ethics | 4 блока среды + распределение, пилот-метрика, онбординг, метрики |
| Рекалибровка | `/recalibration` | reviewer/свои | baseline→30→90→ретро, тренд, рекомендации, история |
| Review | `/review` | reviewer | выбор сотрудника, история, прошлые review, форма, экспорт |
| Подбор | `/hiring` | hr/lead/admin (ethics — чтение) | кандидаты, Quick Screen / Full Calibration, план развития |
| Аудит | `/audit` | admin/ethics | журнал действий |
| Админ | `/admin` | admin | создание/управление пользователями (роли, команды) |

## 5. Экраны: текущее состояние и UX-боли

**Главная** — hero дублирует глобальную шапку (логотип+название встречаются
дважды). Боль: нет ощущения «продукта», карточки-промо статичны.

**Опросник** — длинная вертикальная форма (15 вопросов), радио-шкала 1–5.
Боли: монотонность, нет прогресса/сохранения, шкала не подписана словами на
самих кнопках, мобильная вёрстка тесная. Возможность: пошаговость, прогресс-бар,
подписи концов шкалы.

**Результат опросника** — балл, риск, разбор по компонентам, интерпретация,
дисклеймер. Боль: много текста подряд; разбор по компонентам просится в
визуализацию (бары/радар — но «спокойно», без алармизма).

**Дашборд** — самый плотный экран: 4 блока (среднее + светофор + полоса
распределения), пилот-метрика (headline + 4 блока), онбординг, таблица метрик.
Боли: всё длинной лентой, нет иерархии/группировки, плашка анонимизации
невзрачна, числа без контекста «что хорошо/плохо». Возможность: KPI-карточки,
сетка, мини-тренды, человекочитаемые подписи.

**Рекалибровка / Review / Подбор / Аудит / Админ** — функциональные формы и
списки на сырых Tailwind-классах. Боли: разнородные карточки/кнопки/селекты
(нет единого компонента), таблицы (аудит, метрики) не адаптивны, формы плотные.

**Сквозные боли:** нет единой системы компонентов и состояний (loading/empty/
error выглядят по-разному); типографика плоская (один системный шрифт, мало
иерархии); пустые состояния невыразительны; мобильная адаптивность слабая.

## 6. Приоритеты для дизайна (предложение)

1. **Дизайн-система**: токены (цвет/типографика/спейсинг/радиусы/тени) +
   базовые компоненты (Card, Button, Select/Input, RiskBadge, Distribution bar,
   Disclaimer, EmptyState, StatCard, Table). С соблюдением п.2.
2. **Дашборд** — как флагманский экран: иерархия, KPI-карточки, спокойная
   визуализация риска и распределения.
3. **Опросник + результат** — снизить когнитивную нагрузку, добавить прогресс,
   визуализировать разбор по компонентам «мягко».
4. **Типографика и тёмная тема** — выразительная иерархия, возможно кастомный
   шрифт; сохранить спокойный тон.
5. **Мобайл** — все экраны как минимум usable на узком экране.
6. **Пустые/loading/error состояния** — единый стиль, дружелюбные тексты.

## 7. Тех-контекст и как доводить до прода

- **Стек:** Next.js (App Router) + Tailwind, TypeScript. UI на русском.
- **Где править:** токены — `app/globals.css` + `tailwind.config.ts`; светофор/
  дисклеймеры — `lib/risk.ts`; экраны — `app/<screen>/page.tsx`; шапка —
  `app/layout.tsx`. Компонентов пока нет — создавать в `components/`.
- **Пайплайн:** ветка → PR → зелёный CI → merge в `main` → **автодеплой на прод**
  (`.github/workflows/deploy.yml`). То есть дизайн-правки катятся тем же путём.
- **Не сломать:** бэкенд отдаёт `risk_level`, компоненты, интерпретацию,
  дисклеймеры, флаг `sufficient_data` (анонимизация) — дизайн работает поверх
  этих данных, контракт менять не нужно.
