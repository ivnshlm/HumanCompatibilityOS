# Пилот Human Compatibility OS

Инструкция по запуску пилота MVP (Burnout & Environment Health Monitoring).
Цель пилота — проверить, что система помогает **снизить давление среды**, оставаясь
в рамках доктрины: explainability-first, никаких авто-решений, проверка человеком,
явное согласие, аудит.

## Метрика успеха

**Снижение «Давления аврала» (emergency pressure) ≥ 20% за 90 дней** по команде.

- Headline считается по компоненту `emergency_pressure` (вопросы 1, 5, 11).
- Endpoint: `GET /compliance/pilot-metric/team/{team_id}` — сравнивает среднее
  значение в цикле `baseline` со средним в цикле `day_90`. Возвращает:
  - `headline` — emergency pressure: `pct_change` + `target_met` (порог −20%);
  - `blocks` — изменение всех 4 блоков среды (Burnout Pressure, Recovery
    Sustainability, Communication Entropy, Leadership Stability): `pct_change` и
    флаг `improved` (снижение по шкале «концерна» = улучшение).
- Метрика подавляется при выборке < 3 участников (анти-слежка).

## Жёсткие ограничения (проверяемы)

- `GET /compliance/policy` отдаёт машиночитаемую доктрину
  (`no_automated_decisions=true`, `requires_consent=true`, `requires_human_review=true`).
- Система не содержит эндпоинтов, принимающих кадровые решения, — индикаторы
  только информируют человека.
- Сбор данных невозможен без явного согласия (`POST /questionnaire/submit` → 403
  без согласия).
- Доступ к данным и просмотры аудируются; аудит-лог доступен ролям oversight
  (`admin`, `ethics_reviewer`) через `GET /audit`.

## Подготовка

1. Завести пользователей с ролями: `employee` (участники), `team_lead`/`hr`
   (ревьюеры), `admin` и `ethics_reviewer` (oversight). У участников и
   руководителя одной команды — общий `team_id`.
2. Участники дают согласие (`POST /auth/consent`).
3. Зафиксировать базовую точку: участники проходят опросник, затем создаётся
   событие `baseline` (`POST /recalibration/create`).

## Цикл (на каждого участника)

| День | Действие |
|------|----------|
| 0    | Опросник → событие рекалибровки `baseline` |
| 30   | Опросник → событие `day_30`; на `/recalibration` виден тренд и рекомендации |
| 90   | Опросник → событие `day_90`; считается пилотная метрика по команде |
| Ретро | Опросник → событие `retrospective`; calibration review с источником данных |

## 3–5 пилотных кейсов

1. **Тренд улучшения.** Участник с высоким baseline проходит опросник на 30/90
   днях с более здоровыми ответами → `/recalibration` показывает тренд
   `improving` и Δ < 0. Ожидание: рекомендации соответствуют доминирующему драйверу.
2. **Командная метрика −20%.** Команда из ≥ 3 участников снижает emergency
   pressure baseline → day_90 ≥ 20% → `pilot-metric` отдаёт `target_met=true`.
3. **Подавление малой выборки.** Команда из 1–2 участников → дашборд и пилотная
   метрика возвращают `sufficient_data=false` (никаких индивидуальных данных).
4. **Human review с источником.** HR/Team Lead на `/review` фиксирует
   calibration review с обязательным `source_of_evidence`; участник видит review
   о себе (прозрачность).
5. **Аудит и экспорт.** `ethics_reviewer` через `GET /audit` проверяет, что
   просмотры/изменения залогированы; `GET /export/employee/{id}` выгружает полный
   объяснимый пакет для проверки человеком.

## Что считается провалом пилота

- Любой эндпоинт, который автоматически меняет кадровый статус, — недопустим.
- Появление индивидуальных данных в командных агрегатах при выборке < 3.
- Сбор данных без согласия или без записи в аудит.
